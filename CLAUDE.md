# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`cas-typescript-sdk` is a Node.js cryptography library that wraps the Rust [`cas-lib`](https://github.com/Cryptographic-API-Services/cas-lib) crate (pinned at `=0.2.77` in `Cargo.toml`). The actual cryptographic implementations live in `cas-lib` (which itself builds on RustCrypto and Dalek-Cryptography). This repo contributes two thin layers on top of it:

1. A **Rust napi-rs FFI layer** (`src/`) that exposes `cas-lib` functions to Node as a native addon.
2. A **TypeScript wrapper layer** (`src-ts/`) that provides ergonomic, class-based, documented APIs over the raw FFI functions.

## Build & test commands

```bash
npm run build:rust   # napi build --release --platform → produces native *.node + regenerates index.js / index.d.ts
npm run build:ts     # rimraf lib && tsc → compiles src-ts/ into lib/ (the published JS)
npm run build        # build:rust then build:ts

npm run rust:test    # cargo test --release (runs the inline #[test] fns in src/)
npm run node:test    # npx playwright test (runs the TS tests in tests/)
npm test             # rust:test → build → node:test (the full CI pipeline)
```

- The TS tests import wrappers from `src-ts/`, but the wrappers `require` the native binding from the repo-root `index.js`. **You must run `npm run build:rust` before `npm run node:test`** or the native `.node` binding won't exist. `npm test` handles ordering for you.
- Tests use the **Playwright test runner** (`@playwright/test`) purely as a test framework — there is no browser/page usage despite `playwright.config.ts` defining chromium/firefox/webkit projects. Each test therefore runs three times (once per project). Run `npx playwright install` once before the first `node:test`.
- Run a single TS test file: `npx playwright test tests/symmetric.test.ts`. Filter by title: `npx playwright test -g "aes 128"`. Single project: `npx playwright test --project=chromium`.
- Run a single Rust test: `cargo test --release aes128_encrypt_decrypt_test`.

## Architecture & conventions

**The FFI boundary is code-generated, not hand-written.** `index.js` and `index.d.ts` at the repo root are auto-generated by NAPI-RS (`/* auto-generated by NAPI-RS */`) — never edit them by hand. They are regenerated by `npm run build:rust`. napi converts Rust `snake_case` function names to JS `camelCase` (e.g. Rust `aes128_encrypt` → JS `aes128Encrypt`).

**Data crosses the boundary as `Array<number>` (byte arrays).** Rust signatures use `Vec<u8>`; the TS side passes/receives `Array<number>`. Strings are used only where the underlying API is text-based (RSA PEM keys, password-hash strings). Convert text with `TextEncoder`/`TextDecoder` (see `tests/symmetric.test.ts`).

**The two source trees mirror each other by crypto domain.** Each domain has a Rust module under `src/` and a matching folder under `src-ts/`:

| Domain | Rust (`src/`) | TS (`src-ts/`) |
| --- | --- | --- |
| symmetric (AES) | `symmetric/aes.rs` | `symmetric/` |
| asymmetric (RSA, Ed25519) | `asymmetric/` | `asymmetric/`, `signature/` |
| hashers (SHA, BLAKE2) | `hashers/` | `hashers/` |
| password hashers (Argon2, bcrypt, scrypt) | `password_hashers/` | `password-hashers/` |
| key exchange (X25519) | `key_exchange/` | `key_exchange/` |
| message auth (HMAC) | `message/` | `message/` |
| sponges (Ascon AEAD) | `sponges/` | `sponges/` |
| hybrid (HPKE) | `hybrid/` | `hybrid/` |
| compression (zstd) | `compression/` | `compression/` |

- `src/lib.rs` declares every Rust module — a new module must be registered there.
- `src-ts/index.ts` is a barrel that `export *`s every domain's `index.ts` — a new wrapper must be re-exported up the chain to be part of the public API.
- Wrappers are plain classes that delegate 1:1 to the FFI function and carry the JSDoc (the FFI layer has none). Some domains add a factory + base-interface + enum to select an implementation at runtime — see `src-ts/hashers/` (`HasherFactory` / `IHasherBase` / `HasherType`) and `src-ts/password-hashers/` for the pattern to follow.

**Adding a new algorithm** typically means: (1) write the `#[napi]` function in the right `src/<domain>/*.rs` module (and register the module in `src/lib.rs` if new); (2) run `npm run build:rust` to regenerate `index.d.ts`/`index.js`; (3) add a TS wrapper class in `src-ts/<domain>/` delegating to the new camelCase function; (4) export it through the domain `index.ts` and `src-ts/index.ts`.

**Memory zeroization is enabled globally.** `src/lib.rs` installs `zeroizing_alloc::ZeroAlloc` as the global allocator so freed heap memory is zeroed — relevant when handling key material.

## Inline Rust tests & NIST vectors

Rust modules keep their `#[test]` functions inline in the same `.rs` file (e.g. round-trip encrypt/decrypt checks). Some TS tests validate against official NIST/FIPS known-answer vectors stored under `tests/data/` (AES GCM `.rsp`, SHA-3 `.rsp`); see `tests/aes-fips.spec.ts` and `tests/hasher.spec.ts` for how those `.rsp` files are parsed.

## CI
